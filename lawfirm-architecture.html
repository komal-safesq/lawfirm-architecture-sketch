Executive Summary

The platform's design centers on a single master identity registry (Supabase/PostgreSQL) that sits beneath every downstream tool. Documents, email, calendar, and AI reasoning are all intended to reference this registry rather than each maintain an independent version of the truth. This model is sound and partially implemented: the identity resolution system (FSID) is live and actively used across 378 matter records.

Three components of the intended architecture are designed but not yet operational: the natural-language intake layer, the relationship graph connecting related matters, and the automated time-reconstruction pipeline. Additionally, this assessment identifies four data-integrity and access-control items that should be addressed before further build-out, detailed in Findings below.

None of the items identified here block current operations. They represent prioritized groundwork for the next phase of development.

Findings

Access control is not currently enforced as designed
Row-level security policies exist on the majority of tables but are configured to permit unrestricted access, and a large share of tables are additionally exposed to the anonymous access role. Recommendation: treat all current tables as broadly readable until policies are reviewed and corrected; avoid introducing new confidential data into unreviewed tables in the interim.
Schema history is not version-controlled
The production schema's full migration history exists only in the live database and has not been exported to the source repository. Recommendation: export and commit migration history as the first item of work in the schema repository, prior to further schema changes.
A significant share of built database objects are unregistered or unconnected
Over a hundred tables and schema objects exist without documentation of their purpose or without foreign-key connections to the rest of the system, several of which duplicate the function of tables already in active use. Recommendation: require every new schema object to be documented and connected at creation time; periodically audit and retire unused objects.
AI-assisted queries currently return unverified schema references
A cross-system query test returned table and field names that do not exist in the production database, presented with the same confidence as verified results. Recommendation: do not treat AI-generated schema or data references as authoritative until validated against the live database; any AI-proposed record creation should route through the standard identity-resolution and validation path rather than direct insertion.
