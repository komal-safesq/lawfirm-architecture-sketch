# SafeSQ Engineering Work Log: Classification Guidelines

## Purpose

This document defines how work performed at SafeSQ is recorded and classified. It is the structured, durable record, not a formatted deliverable. Invoices, timesheets, analytics, and any future AI-assisted classification should all be generated *from* this record, not maintained separately from it.

The goal is not simply to log hours. It's to produce documentation that supports:

- Engineering work tracking
- Future analytics
- Potential task taxonomy development
- AI-assisted classification of work
- Consistent project documentation

## Background

The firm's long-term vision is broader than traditional legal billing: a structured engineering task classification system inspired by legal UTBMS, but adapted for engineering work and understandable for day-to-day use.

The taxonomy is **currently under development**. Work log entries should use the classifications below consistently, but should not invent or enforce a finalized UTBMS-style code (e.g. `LX410`, `AD101`) unless explicitly requested.

## Domain Definitions

| Code | Meaning |
|------|----------|
| EDU | Learning, onboarding, technical familiarization |
| ARCH | System architecture |
| DEV | Software development |
| DB | Database engineering |
| AI | Artificial Intelligence |
| OCR | OCR / Document Processing |
| INT | Systems Integration |
| OPS | Operations / Infrastructure |
| RES | Technical Research |

Choose the domain that best represents the primary work in an entry.

## Phase Definitions

| Phase | Meaning |
|--------|---------|
| Learn | Learning new systems or technologies |
| Analyze | Researching existing systems or workflows |
| Design | Planning architecture or schemas |
| Build | Implementing new functionality |
| Enhance | Improving existing systems |
| Integrate | Connecting systems together |
| Validate | Testing and verification |
| Document | Documentation and knowledge capture |

## Activity Examples

- Environment Setup
- System Architecture
- Schema Design
- Database Exploration
- API Integration
- Application Development
- Research
- Testing
- Documentation
- Workflow Analysis
- Prototype Development

Use clear engineering language, not a fixed enum, this list is illustrative.

## Narrative Guidelines

Each work description should:

- Begin with an action verb.
- Explain the engineering objective.
- Mention major technologies when relevant.
- Avoid unnecessary detail.
- Be concise but technically meaningful.

**Good example:** "Developed the initial prototype of the medical coordination platform using HTML, Firebase, and Firestore with real-time synchronization."

**Bad example:** "Worked on website."

## Hours

Hours must:

- Match actual tracked hours.
- Sum exactly to the reporting total.
- Never invent additional work.

## Future Taxonomy

The firm's long-term direction is a UTBMS-like engineering task classification system. Current entries should prepare for that by consistently using Domain, Phase, and Activity as descriptive labels, not official billing codes, until a finalized taxonomy is explicitly adopted.
